# 2026-05-06 issue #537 dynamic dictionary scope

## Goal

Move generated localization templates out of broad global dictionary leaves only
when an owner route is already proven, and record follow-up work for generated
HistorySpice surfaces that still need owner translators.

## Evidence Used

- `docs/RULES.md` owner-route and fixed-leaf policy.
- `Mods/QudJP/Localization/AGENTS.md` dictionary acceptance policy.
- Decompiled Caves of Qud 2.0.4 source under `~/dev/coq-decompiled_stable/`.
- Structural source searches over `XRL.World.Effects`, especially
  `GetDescription()`, `GetDetails()`, `DisplayName = ...`, and `E.AddTag(...)`
  producers.
- Existing QudJP active-effect and display-name translators and L1/L2 tests.
- `XRL.World.Skills.Cooking.CookingRecipe.GenerateRecipeName` and
  `XRL.World.Parts.Campfire` cooking source review for the remaining
  HistorySpice recipe and ingredient templates.

## Implemented Owner Routes

### GetDisplayName state templates

The display-name bracket state templates are generated by effect and part code,
not broad UI leaves. These rows moved from `ui-default.ja.json` into
`Dictionaries/Scoped/ui-displayname-state-templates.ja.json`:

- `sitting on {0}`
- `lying on {0}`
- `enclosed in {0}`
- `engulfed by {0}`
- `auto-collecting {0}`

`GetDisplayNameRouteTranslator` now resolves these through
`GetDisplayName.StateTemplate` and does not call the global `Translator` for the
template key.

### HistorySpice common capture components

`historyspice-common.ja.json` moved out of the top-level dictionary directory
and into `Dictionaries/Scoped/historyspice-common.ja.json`.

The scoped lookup is intentionally narrow:

- `MessagePatternTranslator` uses it for `{tN}` capture components.
- `JournalPatternTranslator` uses it for `{tN}` capture components.
- `HistoricSpiceGeneratedNameTranslator` uses it when reconstructing generated
  dish, festival, and cognomen captures.

Top-level `Translator.Translate("someone")` now remains English unless a
route-specific translator asks for the scoped HistorySpice component dictionary.

## Deferred Follow-ups

### Cooking / HistorySpice recipe names

`world-gospels.ja.json` rows such as `$creaturePossessive $creatureBodyPart`
and `*dishName* ...` remain deferred. `CookingRecipe.GenerateRecipeName` and
`Campfire` meal generation compose these values through HistorySpice expansion,
and `HistoricStringExpanderPatch` is currently disabled to avoid translating
symbolic HistorySpice paths during world generation. Static review of the
cooking producers found the relevant narrow owners, but a correct fix is not a
plain dictionary move:

- `CookingRecipe.GenerateRecipeName` expands one-, two-, and three-ingredient
  recipe templates, then title-cases the final generated name.
- `Campfire.RollIngredients` expands generated ingredient phrases, including
  creature/body-part templates.
- `Campfire.DescribeMeal` expands meal description prose after inserting a
  generated ingredient list.

Moving the raw `world-gospels.ja.json` token templates now would preserve the
data but would not translate the visible cooking output. The safe follow-up is a
dedicated cooking owner translator that operates on final recipe names and meal
ingredient prose without re-enabling broad `HistoricStringExpander` translation.

Follow-up owner work should cover:

- `XRL.World.Skills.Cooking.CookingRecipe.GenerateRecipeName`
- `XRL.World.Parts.Campfire.RollIngredients`
- `XRL.World.Parts.Campfire.DescribeMeal`

Tracked follow-up: <https://github.com/ToaruPen/coq-japanese_stable/issues/538>.
